# 🚚 Apex Logic: ШІ-Диспетчер Транспортної Логістики

**Проєкт розроблено в рамках хакатону BEST::HACKath0n 2026**

**Apex Logic** — це інноваційне програмне рішення для транспортно-логістичних компаній, яке вирішує проблему неефективного розподілу ресурсів та неможливості швидкого перерахунку поставок в умовах динамічного попиту чи форс-мажорів. 

Замість ручного хаосу ми пропонуємо систему "Human-in-the-loop", де ШІ миттєво генерує рішення для водія, а диспетчер лише затверджує його.

📚 **API Документація (Swagger):** [https://apex-logic.online/docs](https://apex-logic.online/docs)

## 🎯 Як ми закрили вимоги MVP

Наш додаток повністю відповідає основним вимогам хакатону:
1. **Зручність у складських приміщеннях та адаптивність:** Фронтенд побудовано на Tailwind CSS + HTMX, що забезпечує миттєве оновлення станів без перезавантаження сторінки та ідеально працює як на ПК диспетчера, так і на мобільному телефоні водія.
2. **Алгоритм пріоритизації:** Ми впровадили систему динамічних статусів (`NORMAL`, `ELEVATED`, `CRITICAL`), які автоматично змінюються ШІ залежно від рівня потреби та типу вантажу (наприклад, пріоритет для медикаментів).
3. **Реагування на термінові запити:** Завдяки інтеграції з OpenAI, система здатна за секунди обробити повідомлення водія (напр. "Дорогу розмило"), перевірити найближчі доступні склади та запропонувати перенаправлення.
4. **Захист інформації:** Усі критичні запити захищені JWT-авторизацією (з хешуванням bcrypt), а обмін даними йде через налаштований HTTPS.

## 🛠 Технологічний стек

* **Backend:** Python 3.10, FastAPI
* **Database:** PostgreSQL, SQLAlchemy (Async), Alembic (Міграції)
* **AI Engine:** OpenAI API (`gpt-4.1-mini`) із застосуванням Structured Outputs (Pydantic) для гарантованої валідації JSON-контрактів
* **Frontend:** HTMX, Tailwind CSS
* **Infrastructure:** Docker, Docker Compose, розгортання на Hetzner, Nginx (Reverse Proxy)

## 🔄 Архітектура: Як це працює (Use Cases)

1. **Ініціалізація:** Диспетчер створює рейс, система перевіряє наявність та автоматично бронює товар (Math Engine виконує UPDATE залишків). Рейс створюється зі статусом `PENDING`.
2. **Прийняття рейсу водієм:** Водій бачить нове призначення у своєму інтерфейсі та натискає кнопку "Почати". Лише після цього статус рейсу офіційно змінюється на активний (`IN_PROGRESS`).
3. **Форс-мажор:** Водій тисне червону кнопку та описує проблему. Ендпоінт `/api/driver/orders/{order_id}/force_majeure` відправляє Context JSON до OpenAI.
4. **AI Рішення:** Модель аналізує ситуацію і повертає `ai_proposed_action` (`REROUTE`, `RESTOCK`, `WAIT`) та розраховану дельту запасів.
5. **Апрув (Human-in-the-loop):** Диспетчер бачить пропозицію ШІ та цифри перерахунку. Натискання "Approve" транзакційно оновлює базу (маршрути, статуси та `inventory`).

## 💻 Інструкція із локального запуску

Проєкт повністю контейнеризований і налаштований на автоматичний запуск. Вам не потрібно вручну створювати БД чи таблиці.

### 1. Клонування репозиторію
```bash
git clone https://github.com/Lert-64/best-hackathon-2026-Apex_logic-test
cd apex-logic
```

### 2. Налаштування змінних оточення
Скопіюйте файл прикладу і додайте туди ваш ключ OpenAI:
```bash
cp .env.example .env
```
Відкрийте `.env` і переконайтеся, що там є валідний `OPENAI_API_KEY`.

### 3. Запуск через Docker
```bash
docker-compose up --build
```
*Alembic автоматично створить таблиці, а скрипт seed.py наповнить базу тестовими складами, залишками та водіями.* Додаток буде доступний за адресою: `http://localhost:8000`.

## 📡 Специфікація API Ендпоінтів

Наш API розділений на логічні блоки з відповідним розмежуванням прав доступу (Role-Based Access Control).

### 🔐 Авторизація
* **`POST /api/auth/login`** Приймає логін та пароль, повертає JWT-токен та роль користувача (`DRIVER` або `DISPATCHER`) для розділення інтерфейсів.

### 🚚 Ендпоінти Водія (Driver API)
* **`GET /api/driver/orders/current`** Дозволяє водію отримати дані про свій поточний активний рейс.
* **`POST /api/driver/orders/{order_id}/accept`** Водій підтверджує прийняття рейсу, статус замовлення змінюється на `IN_PROGRESS`.
* **`POST /api/driver/orders/{order_id}/force_majeure`** **Тригер ШІ-модуля.** Приймає текстовий коментар водія про інцидент, викликає ШІ для аналізу ситуації та створює AI-драфт із пропонованим рішенням (`REROUTE`, `RESTOCK`, `WAIT`).
* **`POST /api/driver/orders/{order_id}/complete`** Фіналізація рейсу. Змінює статус на `COMPLETED` та зараховує фізичний вантаж на склад прибуття.

### 🏢 Ендпоінти Диспетчера (Dispatcher API)
* **`GET /api/dispatcher/inventory`** Отримання глобального зведення щодо фізичних залишків на всіх 8 складах.
* **`GET /api/dispatcher/orders`** Виведення списку всіх рейсів для моніторингу статусу доставки.
* **`POST /api/dispatcher/orders`** Ручне створення нового рейсу диспетчером із миттєвим резервуванням необхідного об'єму товару зі складу відправлення.
* **`PUT /api/dispatcher/orders/{order_id}`** Ручне оновлення маршруту диспетчером (зміна точки прибуття або пріоритету) без залучення ШІ.
* **`POST /api/dispatcher/orders/{order_id}/approve_ai`** Затвердження пропозиції від ШІ. Транзакційно оновлює базу даних (маршрут, пріоритет) та фіксує нові залишки.
* **`POST /api/dispatcher/orders/{order_id}/reject_ai`** Відхилення пропозиції ШІ диспетчером та очищення AI-драфту з поверненням до попереднього плану.

## 🚀 Масштабованість (Плани на майбутнє)

Наша база даних (таблиця `Order`) спроєктована "на виріст". Ми реалізували поле `ai_proposed_delta`, яке містить математичні розрахунки залишків. Це дозволяє в майбутньому:
1. Інтегрувати Google Maps API для точного розрахунку втрат часу та палива.
2. Реалізувати додаткове завдання щодо працездатності в умовах поганого зв'язку шляхом кешування маршрутів на клієнті через Service Workers (PWA).
